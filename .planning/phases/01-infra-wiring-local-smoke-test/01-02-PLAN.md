Output: A ~150–250 line Python script at `scripts/smoke-liveavatar.py`, and an operator-observed run producing a markdown summary ready to paste into Plan 03's rollback-runbook "Sandbox Behavior Baseline" appendix.

**File header (mandatory per CLAUDE.md + CONVENTIONS.md):**

```python
from __future__ import annotations

"""Phase 1 smoke test for the LiveAvatar free-tier/sandbox switch.

Drives the running local FastAPI backend's /session, /speak, and /interrupt
endpoints under the sandbox configuration and prints a paste-ready markdown
summary block mapping SMOKE-01 through SMOKE-04 to PASS / FAIL / OBSERVED.

Prerequisites (see setup-local.ps1):
  1. Backend running on http://localhost:8000 with these env vars set:
       LIVEAVATAR_API_KEY         = <existing paid key — intentional per D-08>
       LIVEAVATAR_AVATAR_ID       = dd73ea75-1218-4ef3-92ce-606d5f7fbc0a
       LIVEAVATAR_IS_SANDBOX      = true
       LIVEAVATAR_SESSION_MODE    = LITE
  2. Python venv with httpx (reuse backend/requirements.txt venv).

Usage:
    python scripts/smoke-liveavatar.py [--base-url http://localhost:8000] [--json]

Exits non-zero if SMOKE-01 fails (hard gate). SMOKE-02 ws_url absence is
reported as OBSERVED, not FAIL (per REQUIREMENTS.md SMOKE-02 degradation
acceptance clause). SMOKE-03 and SMOKE-04 are OBSERVED-only (recorded, not
asserted) per D-07.
"""
```

**Structure (top-to-bottom):**

1. **Imports** — stdlib only + `httpx`: `argparse`, `asyncio`, `json`, `logging`, `sys`, `time`, `uuid`, `dataclasses.dataclass`, `dataclasses.field`, `typing.Any`, `httpx`.

2. **Constants block:**
   ```python
   DEFAULT_BASE_URL = "http://localhost:8000"
   ALLOWED_BASE_URL_PREFIXES = ("http://localhost:", "http://127.0.0.1:")  # SSRF guard — see threat model
   SANDBOX_CAP_WAIT_SECONDS = 130  # slightly above documented 60-120s cap
   CONCURRENT_SESSION_COUNT = 2  # per D-07 (e)
   REQUEST_TIMEOUT_SECONDS = 25.0  # generous — /session can take 5-15s on cold start
   SPEAK_TEXT = "Smoke test: the quick brown fox jumps over the lazy dog."
   ```

3. **Logging setup** matching backend style (`%(asctime)s  %(levelname)-8s  %(name)s — %(message)s`). Use `logger = logging.getLogger("smoke")`. Never log the `X-Session-ID` value together with an auth-like header (defense-in-depth).

4. **`@dataclass` `CheckResult`** with fields: `id: str` (e.g., `"SMOKE-01"`), `status: str` (one of `PASS`, `FAIL`, `OBSERVED`), `summary: str`, `details: dict[str, Any] = field(default_factory=dict)`.

5. **`async def _validate_base_url(url: str) -> None`** — raises `ValueError` if `url` does not start with any of `ALLOWED_BASE_URL_PREFIXES`. This is an SSRF guard: the smoke script is allowed to hit the operator's localhost only.

6. **`async def _preflight(client, base_url) -> None`** — calls `GET {base_url}/health`; raises if non-200. Human-readable error if connection refused ("Is the backend running? See setup-local.ps1.").

7. **`async def run_smoke_01(client, base_url) -> CheckResult`** —
   - Generate `session_id = str(uuid.uuid4())`.
   - `t0 = time.monotonic()`; call `GET {base_url}/session` with header `X-Session-ID: <session_id>`.
   - Capture: HTTP status, elapsed milliseconds (`int((time.monotonic()-t0)*1000)`), parsed JSON body.
   - PASS if status == 200 AND the response body contains a truthy `session_id` field inside `liveavatar_data` (or at top-level, depending on SessionResponse shape — the script should handle both by probing `body.get("session_id") or body.get("liveavatar_data", {}).get("session_id")`).
   - FAIL otherwise. On 4xx with `"is_sandbox"` or `"sandbox"` mentioned in the body, add to `details` the note "Possible D-10 case (a): provider rejected is_sandbox on paid-tier key."
   - NEVER include the `X-Session-ID` header value in `details` if the call failed with 401/403 (defense against accidental secret-like-value leak in copy-pasted summaries).

8. **`async def run_smoke_02(client, base_url, session_id) -> CheckResult`** —
   - Status `OBSERVED` regardless (not a hard gate).
   - Re-read the same `/session` call result's body; record whether `ws_url` is present and non-empty under either `body["ws_url"]` or `body["liveavatar_data"]["ws_url"]`.
   - If present: `summary = "ws_url present in LITE sandbox — full TTS path works"`.
   - If absent: `summary = "ws_url absent in LITE sandbox — degraded to 'avatar visible, no TTS push' path (documented acceptable)"`.

9. **`async def run_smoke_03(client, base_url, session_id) -> CheckResult`** — end-to-end Q&A probe:
   - `POST {base_url}/speak` with body `{"text": SPEAK_TEXT}` and the same session header. Status `OBSERVED`.
   - Record status and `status` field from response (`"ok"` vs `"mock"`).
   - `POST {base_url}/interrupt` with the same session header; record status.
   - Sleep `SANDBOX_CAP_WAIT_SECONDS` (use `asyncio.sleep`) and then call `GET {base_url}/session` again with a **fresh** `X-Session-ID` UUID; record whether the second call returns 200 (clean reconnect after cap).
   - `summary` summarises the three observations on one line.

10. **`async def run_smoke_04(client, base_url) -> CheckResult`** — concurrency probe:
    - Fire `CONCURRENT_SESSION_COUNT` (=2) parallel `GET /session` calls with distinct fresh UUIDs via `asyncio.gather(..., return_exceptions=True)`.
    - Record for each: status code, whether it returned `session_id`, elapsed ms.
    - `summary` one of: `"both sessions succeeded (no observed concurrency cap)"`, `"second session 409ed"`, `"second session silently overrode first"`, or `"unexpected pattern: <detail>"`.
    - Status `OBSERVED`.

11. **`def emit_summary(results: list[CheckResult], *, as_json: bool) -> str`** — returns a markdown block matching this EXACT structure (the operator will copy-paste this into `docs/rollback.md` under Plan 03):

```
## Sandbox Behavior Baseline (observed YYYY-MM-DD)

Backend: http://localhost:8000
Config: LIVEAVATAR_IS_SANDBOX=true, LIVEAVATAR_SESSION_MODE=LITE,
        LIVEAVATAR_AVATAR_ID=dd73ea75-1218-4ef3-92ce-606d5f7fbc0a,
        API key: existing paid key (per Phase 1 decision D-08)

| ID       | Status   | Summary |
|----------|----------|---------|
| SMOKE-01 | PASS     | /session returned 200 in 2143 ms with session_id |
| SMOKE-02 | OBSERVED | ws_url present in LITE sandbox — full TTS path works |
| SMOKE-03 | OBSERVED | speak: ok; interrupt: ok; post-cap reconnect: 200 |
| SMOKE-04 | OBSERVED | both sessions succeeded (no observed concurrency cap) |

Elapsed: <total wall-clock>
```

   - The `YYYY-MM-DD` must be today's date via `datetime.date.today().isoformat()`.
   - If `as_json` is true, print JSON instead of markdown (operator preference — both shapes supported per D-14 / Claude's Discretion section).

12. **`async def main() -> int`** — argparse (`--base-url`, `--json`), validate base URL, preflight, then run SMOKE-01 → 02 → 03 → 04 sequentially (SMOKE-04 must NOT run in parallel with 01-03 because we want the concurrency probe clean). Print the summary block to stdout. Return `0` if SMOKE-01 is PASS, `1` otherwise. SMOKE-02/03/04 results never affect exit code.

13. **Entry point:**
   ```python
   if __name__ == "__main__":
       sys.exit(asyncio.run(main()))
   ```

**Scope discipline (D-13):** do NOT import from `backend.*`; do NOT patch any backend file; do NOT create any file outside `scripts/smoke-liveavatar.py`. Do NOT add any new Python dependency — use stdlib + `httpx` only. Do NOT print the `LIVEAVATAR_API_KEY` or any header value anywhere, even on error.

**Error-handling style** matches backend conventions: broad `except Exception as exc:` allowed at the top of each `run_smoke_XX`, log with `exc_info=True` via `logger.exception`, return a `CheckResult` with `status="FAIL"` and `summary="unexpected error: %s" % type(exc).__name__` (NEVER the exception message, which may contain headers).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Smoke script process → local FastAPI backend (HTTP) | Script runs on operator's machine; the backend is already running on localhost. Script sends only a self-generated UUID in `X-Session-ID`; no user input crosses from outside the machine. |
| Smoke script stdout → operator clipboard → docs/rollback.md | The summary block pasted into the runbook must not contain secret-like values. |
| Operator's shell environment → smoke script | `LIVEAVATAR_API_KEY` lives in `backend/.env`; the smoke script never reads it (it hits the backend, which reads it). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Information disclosure | Smoke script summary output pasted into `docs/rollback.md` could contain tokens from the /session response body (e.g., `ws_token`, `session_token`). | mitigate | Script's `emit_summary` emits only status codes, elapsed times, and textual categories — NEVER the raw body. `CheckResult.details` is used for internal logic only; the markdown summary explicitly lists allowed fields. |
| T-02-02 | SSRF | `--base-url` argparse flag could be pointed at an attacker-controlled URL (e.g., metadata service), causing the script to POST semi-sensitive payloads (though no real secrets are in play). | mitigate | `_validate_base_url` restricts to `http://localhost:` / `http://127.0.0.1:` prefixes; ValueError raised otherwise. |
| T-02-03 | Information disclosure | Broad `except Exception as exc: logger.exception(...)` could log provider response bodies containing tokens if the backend echoed them. | mitigate | Per action section: on exception, the `CheckResult.summary` uses `type(exc).__name__` only (never `str(exc)`); full traceback goes to logger at DEBUG level (script default is INFO). |
| T-02-04 | Tampering | Smoke script exits 0 falsely if SMOKE-01 fails silently | mitigate | Hard gate: script returns 1 unless `results[0].status == "PASS"`. |
| T-02-05 | Information disclosure | `X-Session-ID` header value logged alongside an auth error could be mistaken for a credential (it is not — it's a UUID v4, operator-generated). | accept | UUIDs are not secrets; logging them is benign and helps correlate with backend logs. |
| T-02-06 | Repudiation | Operator pastes an unreviewed summary into the runbook containing an unexpected field (e.g., the script was edited maliciously between review and run). | accept | Script is plain Python in a Git-tracked file; `git diff` before running is the operator's recourse. |

Security enforcement note: all threats are low severity (tooling phase, localhost-only, no production code touched). Highest real risk is T-02-01 / T-02-03 (token leakage via summary copy-paste) — mitigated by the explicit allowlist of fields in `emit_summary`.
</threat_model>
