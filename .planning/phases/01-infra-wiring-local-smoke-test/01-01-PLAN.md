<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub repo vars/secrets → GitHub Actions runner | Non-sensitive config (`vars.*`) is world-readable to anyone with repo read; secrets (`secrets.*`) are masked in logs. |
| GitHub Actions runner → Terraform (TF_VAR_* env) | TF_VAR_* values flow into Terraform state stored in `rg-aicv-tfstate` blob container (Azure RBAC-gated). |
| Terraform state → Azure Container App env | Container App env vars are visible to anyone with `Microsoft.App/containerApps/read` on the RG. |
| Container App env → backend Python runtime | `os.getenv(...)` in `backend/config.py`; no user input reaches these vars. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Information disclosure | `LIVE_AVATAR_IS_SANDBOX` value in Terraform state / CI logs | accept | Not a secret — boolean flag indicating tier; publicly inferable from the rendered avatar. State file is already in Azure Blob Storage behind Azure RBAC; no additional mitigation required. |
| T-01-02 | Information disclosure | `LIVEAVATAR_API_KEY` (existing, untouched by this plan) co-located in the same env block group | mitigate | Existing — API key is stored via `secret { }` / `secret_name = "liveavatar-api-key"`. This plan only ADDS non-secret env vars; it does NOT move the API key out of secrets or change its handling. |
| T-01-03 | Tampering | Someone with repo-variable write access flips `LIVE_AVATAR_IS_SANDBOX` to `false` silently | mitigate | Rollback runbook in Plan 03 documents the legitimate tier-flip procedure; an unauthorized flip would be caught by Phase 2 DEPLOY-02 probe (LiveAvatar dashboard credits > 0). Fallback default `'true'` ensures GitHub-variable absence cannot regress to paid tier. |
| T-01-04 | Spoofing / elevation | Injection of malicious value into `TF_VAR_live_avatar_session_mode` | mitigate | Terraform `validation { }` block constrains values to `LITE | FULL | CUSTOM` — `terraform plan` fails on any other value, including Bash-injection attempts. |
| T-01-05 | Denial of service | Workflow reads `vars.LIVE_AVATAR_IS_SANDBOX` as string `'false'` (lowercase) — backend correctly parses; risk only if GitHub UI forces capitalization inconsistency | accept | `backend/config.py:60` already does `.lower() == "true"`, so `"TRUE"`, `"True"`, `"true"` all work. No CI-level validation needed. |
| T-01-06 | Repudiation | Workflow run history may not capture which variable values were used at deploy time | accept | GitHub Actions logs record the rendered `TF_VAR_*` values for non-secret vars; Terraform state records the applied `env {}` blocks. Audit trail sufficient. |

All threats are low-severity config-surface risks appropriate to a tooling phase.
</threat_model>
